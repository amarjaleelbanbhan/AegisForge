# cortexward-vcs

`VCSPort` adapters (MPS §17.1, §21): a host-agnostic view over `checkout`, `read_diff`,
`open_pull_request`, and `comment`, so orchestration code is identical whether the target
lives on GitHub, GitLab, Bitbucket, Azure DevOps, or disk.

## What exists

- **`GitHubVCSAdapter`** — GitHub's REST API v3 via `urllib.request` (no `PyGithub` dependency),
  plus a real `git` subprocess for `checkout` (GitHub's API has no clone endpoint). Registered
  under the `cortexward.vcs` entry-point group as `github`.

```python
from cortexward.vcs import GitHubVCSAdapter

vcs = GitHubVCSAdapter(token="ghp_...")  # a PAT or an exchanged GitHub App installation token
pr = vcs.open_pull_request(
    repository_url="https://github.com/owner/repo",
    base_branch="main",
    head_branch="cortexward/fix-sql-injection",
    title="Fix SQL injection in query builder",
    body="Generated by CortexWard's RepairAgent; see attached evidence.",
)
vcs.comment(pr, "Rescan confirms this patch is clean.")
```

## Not live-verified

This environment has no GitHub token with write access to a real repository. The request/
response mapping is written against GitHub's published REST API v3 schema and unit-tested
against that documented shape (deterministic, no network) — the same caveat
`AnthropicAdapter`/`GeminiAdapter` (`cortexward-llm`) carry, and for the same reason. Validate
against a real account before depending on this in production.

## Deliberately not implemented

- **GitLab, Bitbucket, Azure DevOps adapters** — `VCSPort` supports them; only GitHub is built.
- **A GitHub App** (JWT + installation-token exchange, a webhook receiver reacting to
  `pull_request`/`push` events, automated PR review comments end to end). `GitHubVCSAdapter`
  accepts a single bearer token and doesn't care whether it's a personal access token or an
  already-exchanged installation token — the exchange itself, and the webhook infrastructure
  that would trigger it, are separate, larger pieces of work. Registering an actual GitHub App
  is an owner-account action (naming, requested permission scopes, public/private, webhook
  secret provisioning) this project can't make unilaterally.
